# Order MVP Implementation Plan

> **For agentic workers:** This document is the source of truth for the first order-service implementation. Do not expand scope during implementation. The first version only supports checkout from the current user's cart.

**Goal:** Implement the first Java backend order flow so an authenticated user can create an `UNPAID` order from selected cart SKU items. The order flow must recompute prices on the server, write immutable order item snapshots, lock inventory with an atomic SQL update, and remove the purchased SKU items from the user's cart after the order is created.

**Architecture:** Continue the existing modular monolith style. The `order` module uses `api`, `service`, `mapper`, `model`, and `dto` packages. `OrderController` exposes public user APIs under `/api/orders`. `OrderService` owns transaction boundaries and business rules. `OrderMapper` uses MyBatis XML for `sales_order` and `order_item`. Inventory locking is added to the existing `InventoryMapper`. Cart cleanup is done through a cart module method scoped by `userId + skuIds`.

**Current dependency boundary:** This phase depends on the existing `cart_item`, `product_spu`, `product_sku`, `category`, `color`, `size_option`, and `inventory` tables. It does not depend on Python AI services.

## 2. Database Design

Create the next Flyway migration after `V5__cart_schema.sql`.

### 2.1 `sales_order`

```sql
CREATE TABLE sales_order (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(64) NOT NULL,
    user_id BIGINT NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL,
    status VARCHAR(32) NOT NULL,
    paid_at DATETIME(6) NULL,
    created_at DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    updated_at DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    UNIQUE KEY uk_sales_order_no (order_no),
    KEY idx_sales_order_user_created (user_id, created_at),
    KEY idx_sales_order_user_status (user_id, status),
    CONSTRAINT fk_sales_order_user FOREIGN KEY (user_id) REFERENCES user_account(id)
);
```

Status values:

- `UNPAID`: created and waiting for payment.

Reserved for future phases:

- `PAID`: payment completed.
- `CANCELLED`: manually cancelled before payment.
- `CLOSED`: timeout closed before payment.

`paid_at` is included now so the later mock-payment phase can update the same table without another schema redesign.

### 2.2 `order_item`

```sql
CREATE TABLE order_item (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id BIGINT NOT NULL,
    sku_id BIGINT NOT NULL,
    spu_id BIGINT NOT NULL,
    sku_code VARCHAR(64) NOT NULL,
    spu_code VARCHAR(64) NOT NULL,
    product_name VARCHAR(128) NOT NULL,
    category_name VARCHAR(64) NOT NULL,
    color VARCHAR(32) NOT NULL,
    size VARCHAR(16) NOT NULL,
    sale_price DECIMAL(10,2) NOT NULL,
    quantity INT NOT NULL,
    line_amount DECIMAL(10,2) NOT NULL,
    main_image_url VARCHAR(512) NULL,
    created_at DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    KEY idx_order_item_order (order_id),
    KEY idx_order_item_sku (sku_id),
    CONSTRAINT fk_order_item_order FOREIGN KEY (order_id) REFERENCES sales_order(id) ON DELETE CASCADE
);
```

Snapshot rule:

- `order_item` is a transaction snapshot, not a live product view.
- Product name, category, color, size, image, SKU code, SPU code, and `sale_price` must be copied from the database at order creation time.
- Later product price changes, product renames, image changes, or product status changes must not alter historical order item display.

### 2.3 Inventory Lock SQL

Add to the existing inventory data access layer:

```sql
UPDATE inventory
SET available_stock = available_stock - #{quantity},
    locked_stock = locked_stock + #{quantity},
    updated_at = CURRENT_TIMESTAMP(6)
WHERE sku_id = #{skuId}
  AND available_stock >= #{quantity}
```

Business meaning:

- Order creation locks inventory.
- Payment success later confirms sale by moving `locked_stock` to `sold_stock`.
- Cancellation or timeout later releases locked inventory back to `available_stock`.

Do not call this "final stock deduction" in comments or docs. It is a lock.

## 6. Service Flow

`OrderService#createOrder` must be `@Transactional`.

Strict execution order:

1. Parse `CurrentUser.userId()` from Controller and pass it into Service.
2. Validate request shape.
3. Reject any `source` other than `CART`.
4. Normalize and de-duplicate `skuIds`.
5. Query selected checkout items from the current user's cart using `userId + skuIds`.
6. If returned row count is lower than requested unique SKU count, return not found because at least one SKU is not in the user's cart.
7. Validate every checkout item has positive cart quantity.
8. Validate SPU and SKU are still `on_sale`.
9. Calculate each `lineAmount = salePrice * quantity`.
10. Calculate `totalAmount` by summing line amounts.
11. For each item, call inventory lock SQL.
12. If any lock returns `0`, throw an insufficient stock error and let the transaction roll back.
13. Generate `orderNo`.
14. Insert `sales_order` with status `UNPAID`.
15. Build `OrderItem` snapshots from checkout items.
16. Batch insert order items.
17. Remove purchased SKU items from the current user's cart.
18. Return `OrderResponse`.

Transaction guarantee:

- If order insert fails, inventory locks roll back.
- If order item insert fails, inventory locks and order header roll back.
- If cart cleanup fails, inventory locks and order inserts roll back.
- If stock is insufficient for any one SKU, all previous SKU locks in the same order roll back.

## 7. Order Number Rule

Recommended first version:

```text
ORD + yyyyMMddHHmmssSSS + 6 random digits
```

Example:

```text
ORD20260602093000123456
```

Rules:

- Do not use database auto-increment ID as the public order number.
- Do not include `userId` in the order number.
- Add unique index `uk_sales_order_no`.
- If insert fails due to rare order number collision, the Service can retry generation a small fixed number of times, such as 3.

Snowflake ID is reserved for a later scale-focused phase. Do not introduce it in the MVP.

## 8. Error Handling

Recommended errors:

- Empty `skuIds`: `BadRequestException("skuIds must not be empty")`
- Unsupported source: `BadRequestException("order source is not supported: " + source)`
- SKU not in current user's cart: `ResourceNotFoundException("cart item not found")`
- Product or SKU not on sale: `BadRequestException("sku is not available for checkout: " + skuId)`
- Inventory lock affected rows `0`: `BadRequestException("insufficient stock for sku: " + skuId)`
- Order detail not owned by current user: `ResourceNotFoundException("order not found: " + orderNo)`

If a dedicated `InsufficientStockException` is added, it should still map to a clear client error response through `GlobalExceptionHandler`.

## 11. Future Payment Phase

Do not implement this in Order MVP. Keep it as the next phase.

### Mock Pay API

```http
POST /api/payments/mock-pay
Authorization: Bearer <accessToken>
Content-Type: application/json
```

Request:

```json
{
  "orderNo": "ORD20260602093000123456"
}
```

Payment flow:

1. Query order by `userId + orderNo`.
2. Validate order status is `UNPAID`.
3. Query order items.
4. For each item, convert locked inventory to sold inventory:

```sql
UPDATE inventory
SET locked_stock = locked_stock - #{quantity},
    sold_stock = sold_stock + #{quantity},
    updated_at = CURRENT_TIMESTAMP(6)
WHERE sku_id = #{skuId}
  AND locked_stock >= #{quantity}
```

5. Update `sales_order.status = 'PAID'`.
6. Set `sales_order.paid_at = CURRENT_TIMESTAMP(6)`.

Future cancellation and timeout flow:

```sql
UPDATE inventory
SET available_stock = available_stock + #{quantity},
    locked_stock = locked_stock - #{quantity},
    updated_at = CURRENT_TIMESTAMP(6)
WHERE sku_id = #{skuId}
  AND locked_stock >= #{quantity}
```
