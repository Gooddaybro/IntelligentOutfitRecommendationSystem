# AI Outfit Workbench Visual Upgrade Design

## 目的

本文记录本轮关于前端视觉升级的讨论结论，作为下一步开发前的确认文档。

本次目标不是重写业务逻辑，也不是把系统继续做成普通后台页面，而是把现有前端升级为一个更适合项目展示的 **AI 服装导购工作台**。

核心目标：

- 用户进入页面后，第一眼理解系统能力：输入穿搭需求，AI 推荐商品。
- 页面体现“AI 智能导购 + 高级电商”的产品定位。
- 商品推荐、AI 理解、搭配清单形成完整链路：需求输入 -> AI 理解 -> 商品推荐 -> 加入购物车/搭配清单。
- 第一阶段尽量只修改前端布局、样式和 className，不修改接口、不调整后端业务逻辑。

## 产品定位

推荐定位：

> AI 服装导购工作台：用户输入穿搭需求，AI 理解偏好，推荐商品，并沉淀为搭配清单。

当前页面不应继续强化“后台管理系统”观感。新的视觉方向应更接近一个可直接演示的 AI 导购产品：

- 不是普通商品后台。
- 不是纯聊天机器人。
- 不是普通商城列表页。
- 是以 AI 推荐为核心的服装导购工作台。

## 目标风格

采用：

> 高级电商风 + AI 智能导购感 + 轻微华丽视觉

风格原则：

- 整体精致、克制、清爽。
- 以商品图片和推荐理由作为视觉重点。
- 避免廉价电商感和后台管理系统感。
- 避免过度动画、粒子、霓虹、复杂 3D。
- 通过留白、层级、阴影、圆角、图片比例、按钮权重来提升质感。

建议配色：

```text
页面背景：#F6F4F0 / #F7F7F5
主文字：#1F1F1F
次文字：#777777
主色：#111827
AI 强调色：#0F766E
价格强调：#B45309
边框：#E5E0D8
卡片背景：#FFFFFF
```

颜色使用原则：

- 不使用大量亮蓝、纯绿、纯红。
- 主按钮使用深色或墨绿色，显得更稳。
- 价格使用暖棕色或偏品牌感的强调色。
- AI 匹配、推荐理由使用墨绿色作为语义强调。

## HTML 原型与截图视觉参考

本次开发以用户提供的 HTML 原型和截图作为第一阶段视觉参考。该原型已经基本表达了目标产品气质：白色高端电商页面、米白导购首屏、深色 AI 搜索按钮、墨绿色智能强调、商品大图卡片、右侧搭配清单。

实现时应吸收其视觉语言和页面结构，但不能原样照搬静态 HTML。当前项目已经是 React + Vite 前端，并且有真实接口、购物车状态、AI 流式聊天和确认弹窗。开发时应将原型拆解并映射到现有组件，不引入静态假数据覆盖真实业务数据。

### 可吸收的视觉点

#### 页面整体

参考截图中的整体观感：

- 页面背景使用米白或浅灰，而不是纯白铺满。
- 主内容区域保持干净、明亮、留白充足。
- 卡片边框要轻，避免厚重后台边框。
- 页面看起来像一个完整的 AI 导购产品，而不是临时拼接的管理后台。
- 大面积区域保持克制，不使用过多强色块。

建议效果：

```text
黑白灰基础质感
+ 米白背景
+ 墨绿色 AI 强调
+ 暖棕价格
+ 轻边框和轻阴影
```

#### 顶部导航

参考原型顶部：

```text
Logo: AI Outfit Stylist
导航：首页 / AI 导购工作台 / 商品中心
右侧：用户
```

可吸收点：

- Logo 位于左侧，品牌名清晰。
- 当前导航项使用墨绿色高亮。
- 导航不使用厚按钮，保持轻量文本式导航。
- 顶部区域使用白底和浅边框，形成稳定产品框架。

落地到当前项目时需要保留现有能力：

- AI 推荐
- 传统浏览或商品中心
- 购物车
- 订单
- 当前登录用户
- 退出登录

因此不应把现有导航简化到只剩三个静态链接。可以在视觉上接近原型，但功能入口必须完整。

#### AI 智能导购首屏

参考原型首屏：

```text
让 AI 为你挑选合适穿搭
输入场景、预算、风格，快速生成专属商品推荐

[ 我想找一套适合春天通勤的穿搭，显瘦一点，预算500以内... ] [智能推荐]

上班通勤 / 约会穿搭 / 学生党 / 显高显瘦 / 平价百搭
```

可吸收点：

- 首屏居中，强调 AI 导购核心动作。
- 标题要直白告诉用户这个页面能做什么。
- 输入框要足够大，像 AI 搜索框，而不是普通表单输入。
- 搜索按钮使用深色实心按钮，形成高级感。
- 快捷标签用于降低用户输入门槛。
- 首屏背景使用柔和米白渐变或浅色块。

实现约束：

- Hero 不应只做装饰，输入框最好接入现有 AI 聊天提交逻辑。
- 如果第一阶段为了降低改动风险，Hero 输入框可以和聊天输入共享同一份 draft 状态，或者点击后聚焦到 ChatPanel 输入区。
- 快捷标签可以用于填充输入内容或触发推荐，但不能绕过现有 AI 请求边界。
- 不应在 Hero 中写死用户已经输入过的需求。

高度建议：

- 原型首屏较有仪式感，但实际工作台不能过高。
- 桌面端建议控制在 `260px - 340px`。
- 保证首屏下方至少能露出一部分商品区域，让用户知道推荐结果就在下面。

#### 主体三列布局

参考截图主体：

```text
左侧：AI 需求分析
中间：推荐商品大图卡片
右侧：搭配清单
```

可吸收点：

- 三列结构清楚，产品叙事完整。
- 中间商品区域占最大视觉权重。
- 左侧解释 AI 如何理解用户需求。
- 右侧承接购物车和搭配清单，形成转化路径。

推荐桌面端比例：

```text
左侧：280px - 320px
中间：1fr，主展示区域
右侧：300px - 340px
```

注意：

- 中间商品区不能被两侧挤得太窄。
- 商品图应是中间区域视觉中心。
- 右侧清单需要 sticky 或保持在较醒目位置时，应避免遮挡弹窗、浮动购物车和移动端内容。

#### 商品大图卡片

参考截图中商品卡片的核心效果：

- 图片很大。
- 图片顶部覆盖 `AI 匹配 xx%` badge。
- 卡片圆角明显但不过度。
- hover 时可以轻微上浮。
- 商品卡片之间留白较宽，避免拥挤。

必须保留真实业务信息：

- 商品名称。
- 商品价格。
- 颜色、尺码、版型或库存提示。
- AI 推荐理由。
- 加入购物车。
- 立即购买。

建议卡片结构：

```text
商品图片区域
  - 图片
  - AI 匹配 badge

商品信息区域
  - 商品名
  - 价格
  - 颜色 / 尺码 / 版型
  - AI 推荐理由提示块
  - 加入购物车 / 立即购买
```

数据规则：

- 如果 `rankScore` 存在，显示 `AI 匹配 xx%`。
- 如果 `rankScore` 不存在，可以显示 `AI 推荐`，也可以不显示 badge。
- 不允许为了视觉效果伪造匹配百分比。
- 如果 `recommendationReason` 存在，展示推荐理由提示块。
- 如果 `recommendationReason` 不存在，不写假理由，可以展示更弱的商品属性信息。
- 商品图片必须优先使用接口返回的 `mainImageUrl`。
- 如果没有图片，展示高级占位，不使用随机外链图冒充真实商品。

#### AI 需求分析卡片

参考原型中的绿色浅底识别区域：

```text
已精准识别您的需求：
场景：上班通勤
预算：¥200 - ¥500
风格：简约 / 休闲
策略：优先百搭、舒适、易搭配
```

可吸收点：

- 使用浅绿色背景表达 AI 分析。
- 信息按维度分组，减少纯聊天记录的随机感。
- 左侧区域先展示“当前需求理解”，再展示聊天记录。

实现约束：

- 不应硬编码“已精准识别”之类过度确定的文案，除非后端真的返回结构化识别结果。
- 第一阶段可以使用更稳妥文案，例如：
  - `当前需求线索`
  - `AI 正在理解`
  - `已填写偏好`
  - `等待你的穿搭需求`
- 可以基于现有 filters 展示：分类、风格、季节、预算。
- 可以基于最近一条用户消息展示“最近需求”。
- 不新增 Java 或 Python 接口来生成这些字段。

推荐空状态：

```text
等待你的穿搭需求
输入场景、预算、风格后，AI 会在这里整理你的偏好线索。
```

#### 搭配清单

参考原型右侧：

```text
搭配清单（已选 3 件）
上衣：宽松连帽卫衣  ¥199
下装：直筒休闲牛仔裤 ¥159
鞋子：厚底增高小白鞋 ¥229
```

可吸收点：

- 名称从普通“购物车”升级为“搭配清单”。
- 每个清单项显示小图、名称、价格。
- 价格使用暖棕色，和商品卡价格保持一致。
- 清单用虚线或浅分割线区分项目，视觉轻。

实现约束：

- 底层继续复用现有购物车数据。
- 不新增“一键采纳整套方案”的业务。
- 不强行把商品归类为上衣、下装、鞋子，除非现有数据中有稳定分类。
- 如果购物车数据中没有价格或图片，则保持可用的摘要展示，不伪造。
- 按钮建议使用当前已有流程，例如：
  - `查看购物车`
  - `去结算`
  - `继续完善搭配`

暂不建议使用：

```text
一键采纳此方案
```

原因：当前业务没有真正的整套方案采纳流程，直接使用该文案会造成能力暗示过强。

#### 图标和装饰

原型中使用了少量 `✨`、`💡` 等符号，能传达 AI 感，但正式页面应控制密度。

建议：

- 优先使用现有 `lucide-react` 图标。
- 关键 AI 区域可以保留少量符号或图标强化识别。
- 不要在每个标题、按钮、卡片里都使用 emoji。
- 不使用手写 SVG 复制原型中的图标；已有 lucide 图标时优先使用 lucide。

### 必须润色和收敛的点

#### 不直接照搬静态商品

HTML 原型中的商品、图片、价格和推荐理由是静态示例。实现时必须使用当前接口数据：

- `candidate.mainImageUrl`
- `candidate.name`
- `candidate.salePrice`
- `candidate.color`
- `candidate.size`
- `candidate.fitType`
- `candidate.stockStatus`
- `candidate.recommendationReason`
- `candidate.rankScore`

不能为了达到截图效果，把静态 Unsplash 图片、固定价格、固定商品名写入 React 页面。

#### Hero 输入必须有明确交互边界

Hero 输入框有两种可接受实现：

1. 与现有 ChatPanel 共享输入和提交逻辑，用户在 Hero 中输入后直接发起 AI 请求。
2. Hero 输入作为快捷入口，提交后把内容带入 ChatPanel 并触发同一套提交流程。

不接受：

- Hero 输入框只是装饰，无法提交。
- Hero 触发一套绕开现有 `streamAssistantChat` 的新逻辑。
- Hero 直接伪造推荐商品结果。

#### 商品信息不能为了大图而缺失

截图中的商品图片占比很大，这是正确方向。但当前系统是可交易商城，商品卡仍必须能支持购买决策：

- 商品名要清楚。
- 价格要突出。
- SKU 属性要可见。
- 推荐理由要可读。
- 加购和购买动作要明确。

因此最终商品卡不能只是一张大图加 badge。

#### 移动端必须重新排布

HTML 原型主要是桌面端。React 实现必须补充移动端响应式。

移动端推荐顺序：

```text
AI 智能导购首屏
推荐商品
搭配清单
AI 需求分析和聊天
```

要求：

- 三列改为单列。
- 商品卡图片保持稳定比例。
- Hero 输入框按钮可换行或压缩，不能溢出。
- 快捷标签允许横向滚动或换行。
- 搭配清单不能被浮动购物车遮挡。

### 与当前 React 项目的映射

推荐映射关系：

```text
原型 .navbar
-> frontend/src/app/App.tsx 顶部导航 + frontend/src/styles.css

原型 .hero
-> frontend/src/pages/AiShoppingPage.tsx 新增 AI 智能导购首屏

原型 .main-container
-> frontend/src/pages/AiShoppingPage.tsx 主体布局

原型 .sidebar 左侧
-> frontend/src/features/assistant/ChatPanel.tsx 或 AiShoppingPage 外层需求理解区域

原型 .product-card
-> frontend/src/features/catalog/ProductCard.tsx

原型 .cart-list
-> frontend/src/pages/AiShoppingPage.tsx 右侧搭配清单，底层复用 cartItems
```

第一阶段优先使用现有 CSS 文件完成样式升级，不引入新的大型 UI 框架。

## 页面结构

推荐采用三段式结构：

```text
┌──────────────────────────────────────────────┐
│ 顶部导航：Logo / 首页 / AI推荐 / 商品 / 购物车 / 用户 │
├──────────────────────────────────────────────┤
│ 首屏智能导购区：一句话说明 + AI 输入框 + 场景快捷标签 │
├───────────────┬────────────────────┬─────────┤
│ AI 需求理解区   │ 推荐商品主区域        │ 搭配/购物车 │
│ 对话和偏好分析   │ 商品卡片 / 筛选 / 理由 │ 实时清单    │
└───────────────┴────────────────────┴─────────┘
```

### 顶部导航

目标：从“后台 topbar”调整为“轻量产品导航”。

建议内容：

- Logo：`AI Outfit` 或 `AI 服装导购`
- 导航：首页 / AI 推荐 / 商品中心 / 购物车 / 订单
- 用户入口：用户名 + 退出登录按钮

设计要求：

- 减少厚重边框。
- 保留清晰的当前导航状态。
- 按钮更轻，不要像表单按钮堆在顶部。
- 移动端允许换行或压缩为更紧凑布局。

### AI 智能导购首屏

目标：让用户一进入页面就知道“这里可以输入需求，AI 会推荐商品”。

建议内容：

```text
AI Outfit Stylist
你的智能服装导购助手
输入你的场景、风格、预算，AI 为你推荐合适穿搭

[ 我想找一套适合春天通勤的穿搭……        搜索/发送 ]

上班通勤   约会穿搭   学生党   显瘦   平价   秋冬
```

设计要求：

- 首屏导购区要精致，但不要占满整屏。
- 推荐高度控制在 220px - 320px。
- 使用柔和浅灰、米白或墨绿色渐变。
- 输入框要大，像 AI 搜索框。
- 快捷标签用于引导用户理解可输入内容。
- 可加入商品图片拼贴或视觉氛围区，但第一阶段可先用 CSS 和现有商品图完成。

### 主体三列工作台

桌面端布局：

```text
左侧：AI 需求理解 + 对话区
中间：推荐商品卡片网格
右侧：我的穿搭组合 / 购物车摘要
```

比例建议：

- 左侧：偏窄，承载理解和对话。
- 中间：最大，是视觉主区域。
- 右侧：偏窄，承载搭配清单和结算摘要。

第一阶段可以保留现有组件状态和接口，只调整 DOM 结构、className 和 CSS。

## 模块设计

### 左侧：AI 需求理解

目标：不要只像聊天软件，要让用户感觉 AI 正在分析需求。

推荐结构：

```text
AI 需求理解

你当前的需求：
春季 / 通勤 / 平价 / 显瘦 / 简约

AI 已识别：
场景：上班通勤
预算：200-500
风格：简约休闲
颜色偏好：黑白灰
推荐策略：优先百搭、舒适、易搭配

聊天记录
输入框
```

第一阶段边界：

- 不新增后端结构化识别接口。
- 可基于现有筛选项展示基础标签：分类、风格、季节、预算。
- 如果无法从数据中稳定推断，不硬编码虚假的 AI 识别结果。
- 聊天流式逻辑保持不变。

### 中间：推荐商品主区域

目标：让推荐商品成为页面视觉中心。

商品区域需要比当前更像电商页面，而不是后台列表。

建议能力：

- 商品卡片网格或瀑布流。
- 商品图片更大。
- 卡片层级更清楚。
- 推荐理由和 AI 匹配度突出。
- 加入购物车、立即购买按钮权重明确。

### 商品卡片

推荐结构：

```text
┌──────────────────────┐
│      商品大图          │
│ AI 匹配 92%           │
├──────────────────────┤
│ 黑色宽松连帽卫衣       │
│ ¥199                  │
│ 风格：休闲 / 通勤      │
│ 尺码：M L XL          │
│ AI 推荐理由：          │
│ 适合春季通勤场景，颜色百搭 │
│ [加入购物车] [立即购买] │
└──────────────────────┘
```

设计要求：

- 图片使用稳定比例，建议 4:5 或 1:1。
- 价格视觉权重大于普通描述。
- `AI 匹配 xx%` 使用醒目 badge。
- 推荐理由使用独立浅色提示块，不作为普通段落堆文字。
- `加入购物车` 和 `立即购买` 按钮需要清晰区分主次。
- hover 时可以轻微上浮和增强阴影。

第一阶段数据边界：

- 已有 `rankScore` 时展示匹配度。
- 没有 `rankScore` 时不伪造百分比，可展示 `AI 推荐` 或隐藏匹配值。
- 继续复用现有 `recommendationReason`。
- 不新增商品详情接口。

### 右侧：我的穿搭组合

目标：把普通购物车包装成更符合 AI 导购语境的“搭配清单”。

推荐名称：

- 我的穿搭组合
- 今日搭配清单
- 搭配清单

推荐结构：

```text
我的穿搭组合
已选 3 件

上衣：黑色卫衣 ¥199
裤子：直筒牛仔裤 ¥159
鞋子：白色板鞋 ¥229

总价：¥587
AI 整体评价：简约、协调、适合通勤

[去结算]
```

第一阶段边界：

- 底层继续复用购物车数据。
- 不新增真正的“整套穿搭生成”业务。
- 如果没有稳定分类信息，不强行归类为上衣、裤子、鞋子。
- 总价可由已有购物车项计算；如果当前数据结构不含价格，则保持现有购物车摘要。
- `去结算` 继续走现有购物车/订单流程。

## 移动端布局

移动端不建议保留三列。

推荐顺序：

```text
顶部 AI 输入区
↓
推荐商品
↓
搭配清单
↓
AI 对话
```

原因：

- 手机端用户最先关心推荐商品。
- 聊天记录不应占据首屏过多空间。
- 搭配清单应靠近购买动作。

移动端要求：

- 商品卡片不得文字溢出。
- 导航可换行，但不能互相遮挡。
- 按钮区域保持可点击尺寸。
- 购物车浮层不能挡住主要按钮。

## 第一阶段开发范围

建议第一阶段只做视觉和布局升级，不改业务逻辑。

可修改文件：

- `frontend/src/app/App.tsx`
- `frontend/src/pages/AiShoppingPage.tsx`
- `frontend/src/features/assistant/ChatPanel.tsx`
- `frontend/src/features/catalog/ProductCard.tsx`
- `frontend/src/styles.css`

主要工作：

1. 调整全局视觉变量：背景、主色、强调色、边框、阴影、圆角、按钮、输入框。
2. 优化顶部导航：更轻、更像产品导航。
3. 在 AI 推荐页新增智能导购首屏区域。
4. 将主体布局调整为：AI 需求理解 / 推荐商品 / 搭配清单。
5. 重做商品卡片样式：大图、价格、AI badge、推荐理由提示块、按钮层级。
6. 将右侧购物车视觉包装为“我的穿搭组合”。
7. 完成桌面端和移动端基础响应式适配。

## 暂不开发范围

第一阶段不做：

- 不改 Java 后端接口。
- 不改 Python AI agent。
- 不改 SSE 流式协议。
- 不新增真实“生成完整穿搭方案”业务。
- 不新增商品详情接口。
- 不引入大型 UI 框架。
- 不加入复杂动画、粒子、霓虹、3D。

## 验证方式

开发完成后应验证：

- 前端构建通过。
- AI 推荐页可以正常打开。
- 登录后默认入口仍能进入 AI 推荐页。
- 聊天输入和流式回复不受影响。
- 推荐商品仍能展示。
- 加入购物车和立即购买动作仍走确认弹窗。
- 购物车/搭配清单数量和内容与现有状态一致。
- 移动端布局无明显遮挡、溢出、重叠。

```bash
cd frontend

```
