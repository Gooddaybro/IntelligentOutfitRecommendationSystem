# Elasticsearch 商品索引生命周期设计

## 1. 背景与目标

当前 Java 全量重建已经能够创建时间戳索引、批量写入、校验数量并原子切换 `product_current`。尚缺少两个生命周期闭环：失败的新索引可能残留，成功重建产生的历史索引会无限增长。

本阶段只解决索引清理：

1. 重建失败时删除本次创建的半成品索引；
2. 重建成功并切换别名后，保留当前索引和最近两个历史索引；
3. 删除范围受到严格名称和别名保护；
4. 不实现增量同步、缓存版本化或监控平台。

## 2. 保留策略

配置项 `app.elasticsearch.retained-history-count` 默认值为 `2`，含义是保留两个历史索引，不包含当前别名索引。因此正常状态最多保留三个受管物理索引：

```text
product_current -> product_20260720120000
                   product_20260719120000
                   product_20260718120000
```

索引按 Elasticsearch 返回的创建时间降序排列。当前别名目标无条件保留，其余受管索引保留最新两个，剩余索引才允许删除。

## 3. 删除安全边界

生命周期组件只管理满足以下全部条件的索引：

- 名称以 `app.elasticsearch.index-prefix` 开头；
- 前缀之后严格为 14 位 UTC 时间戳 `yyyyMMddHHmmss`；
- 不是 `product_current` 当前指向的索引；
- 删除候选由 Elasticsearch 实际索引列表产生，而不是由用户请求传入。

实验索引 `product_v1`、Kibana 系统索引以及名称不符合规则的业务索引不会被删除。即使配置前缀错误，严格时间戳校验也会阻止通配删除。

## 4. 组件边界

新增 `ProductSearchIndexLifecycleService`，专门封装：

- 删除指定的新建索引；
- 查询受管物理索引及创建时间；
- 查询当前别名目标；
- 计算并删除超出保留窗口的历史索引。

`ProductSearchIndexService` 仍负责编排重建，只在两个位置调用生命周期组件：

```text
创建新索引后发生失败
  -> best-effort 删除本次新索引
  -> 抛出原始重建异常

别名切换成功
  -> best-effort 执行历史清理
  -> 返回重建成功结果
```

生命周期逻辑与 Bulk、Mapping 和 MySQL 投影分离，避免继续扩大重建服务的职责。

## 5. 错误处理

### 5.1 重建失败后的清理失败

保留原始重建异常作为调用方看到的错误。清理异常只写入警告日志并作为 suppressed exception 附加到原异常，避免真正原因被二次异常覆盖。

只有确认本次已经成功创建物理索引后才尝试失败清理。如果创建索引本身失败，不发送删除请求。

### 5.2 成功后的历史清理失败

别名已经完成原子切换，搜索能力已经成功，因此接口仍返回成功。清理失败写入警告日志，留待下一次成功重建再次尝试。后续 P4 会为此增加指标。

### 5.3 索引不存在

失败清理使用忽略不存在语义，保证重复清理安全。历史清理在列表与删除之间遇到索引被移除时也视为幂等成功。

## 6. 测试与验收

单元测试必须覆盖：

- 当前索引始终保留；
- 只保留最近两个历史索引；
- 不符合严格命名规则的索引不参与删除；
- 重建失败只删除本次创建的索引；
- 创建索引失败时不执行删除；
- 失败清理异常不覆盖原始异常；
- 成功后的清理异常不改变重建成功结果；
- `retained-history-count` 不能为负数。

集成验证必须连续执行至少四次重建，并确认：

```text
product_current 只有一个目标
受管物理索引最多三个
搜索文档数量与 MySQL 投影一致
中文搜索仍能返回结果
```

完整 `mvnw.cmd verify` 必须通过，新增类型和复杂删除逻辑必须提供解释边界与原因的注释。

## 7. 明确不做

- 不通过定时任务独立清理；只在重建流程中触发；
- 不自动回滚别名；历史索引仅提供人工回滚窗口；
- 不删除当前索引；
- 不清理由旧版本遗留但不符合严格命名规则的索引；
- 不在本阶段实现 Outbox、RabbitMQ、缓存版本化或监控面板。
