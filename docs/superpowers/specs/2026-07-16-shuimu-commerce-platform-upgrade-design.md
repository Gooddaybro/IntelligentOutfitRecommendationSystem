# 水木 AI 完整商城与管理后台升级设计

## 1. 文档目的

本文档在 `2026-07-15-shuimu-frontend-structural-redesign-design.md` 的视觉与应用壳决策上继续演进，把当前“AI 穿搭工具 + 商品展示”升级为同时具备完整用户购物链路和独立运营后台的商城系统。

本轮不推翻已经确认的“浅橡木自然”视觉、桌面侧边导航和平板/手机响应式策略。新增工作的重点是补齐四条产品链路：

1. 商品决策：分类、搜索、商品详情、SKU 选择和 AI 建议。
2. 结算履约：购物袋、确认订单、支付结果、订单详情、物流和售后。
3. 用户资产：个人资料、衣橱画像、收藏、地址和账户安全。
4. 后台运营：商品、分类、SKU、库存、订单、用户、分析和操作日志。

## 2. 已确认决策

| 决策 | 结论 |
| --- | --- |
| 产品目标 | 从 AI 穿搭展示应用升级为可完成购买、履约和运营管理的完整商城。 |
| 特色定位 | AI 造型师继续是核心差异化入口，但不能替代标准商城购买路径。 |
| 视觉方向 | 继续使用鼠尾草绿、亚麻白、浅橡木和深森林绿，不重新选择主题。 |
| 用户端导航 | 桌面保留左侧导航，手机使用五项底部导航。 |
| 管理端边界 | 使用独立 `/admin/*` 路由和独立管理台壳层，不并入用户端页面。 |
| 开发方式 | 契约驱动、前端优先；先完成可演示的全前端，再逐阶段扩展 Java、数据库和真实接口。 |
| 数据真实性 | 前端演示数据只能存在于显式 Mock 模式；生产/API 模式不得伪造价格、库存、订单、物流或用户事实。 |
| 后端事实源 | Java 后端继续作为用户、商品、SKU、价格、库存、订单、支付和后台权限的唯一可信来源。 |

## 3. 范围与阶段边界

### 3.1 本轮包含

- 用户端完整信息架构和响应式应用壳。
- 首页、AI 造型师、商品列表和商品详情。
- 购物袋、确认订单、支付结果、订单列表和订单详情。
- 简化物流信息、取消订单、确认收货和售后申请入口。
- 个人资料、衣橱画像、收藏、收货地址和账户安全。
- 独立管理员登录边界与后台应用壳。
- 后台数据概览、商品、分类、SKU/库存、订单、用户、经营分析和操作日志页面。
- 前端 Mock/API 双适配层及新增接口契约。
- 后续 Java API、数据库迁移、权限、审计和联调方案。

### 3.2 暂不包含

- 第三方真实支付渠道、真实短信和邮件服务。
- 对接外部物流平台；首期物流使用后台维护的承运商、运单号和节点摘要。
- 复杂优惠券、会员等级、积分、秒杀和促销规则引擎。
- 自动退款打款；售后首期记录申请与处理状态。
- AI 虚拟试穿、3D 服装、搭配收藏和社交分享。
- 多商户、供应商结算和多仓库调拨。
- 管理员自定义报表与复杂 BI 平台。

页面可以为暂缓能力保留稳定扩展位，但不能显示成已经可用的操作。

## 4. 总体开发策略

采用“前端全景先行，后端按契约落位”的两段式路线。

### 4.1 第一段：先完成前端效果

前端新增统一的领域类型和仓储接口：

```text
页面 / hooks
  -> commerce repositories
      -> Mock repositories（演示模式）
      -> HTTP repositories（真实 API 模式）
```

- `VITE_DATA_MODE=mock`：使用固定、可重复、覆盖正常/空/错状态的本地演示数据。
- `VITE_DATA_MODE=api`：调用 Java API；这也是正式构建默认模式。
- Mock 模式在界面显示“演示数据”标记，防止把演示价格、库存和订单误认为真实事实。
- 页面不直接判断数据来源，所有切换都在仓储装配层完成。
- Mock 与 HTTP 实现共享同一 TypeScript 契约测试，降低后续联调返工。

### 4.2 第二段：后端逐模块替换 Mock

后端按用户链路和管理能力分批实现。每批完成后，只替换对应仓储实现，不重写页面组件。

```text
前端已完成页面
  -> 固定请求/响应契约
  -> Java Controller / Service / Mapper
  -> MySQL 迁移与真实数据
  -> 契约测试
  -> 切换到 HTTP repository
```

```text
/app/profile/favorites            我的收藏
/app/profile/addresses            收货地址

```

桌面左侧导航：

```text

今日
```

商品详情、确认订单、支付结果和订单详情属于流程页，通过面包屑、返回入口和上下文操作进入，不占用一级侧边导航。

手机底部导航固定为：

```text

```

订单、衣橱画像、收藏、地址和安全设置统一收进“我的”。

```text

/admin/orders                     订单管理
```

## 6. 用户端页面设计

### 6.1 今日首页

首页从“用户状态摘要”升级为商城与 AI 的共同入口：

- 顶部问候、全局搜索和 AI 造型师主入口。
- 常用分类入口：上装、外套、裤装、裙装、鞋靴等，以真实启用分类为准。
- 今日推荐商品，优先使用真实推荐结果；无推荐时使用可解释空状态。
- 最近浏览、收藏、购物袋和待处理订单入口仅在有真实数据时出现。
- 保留衣橱画像完成度和偏好摘要，但不抢占商品与 AI 主任务。

首页不展示无法从后端获得的天气、虚构销量或虚构优惠。

### 6.2 AI 造型师

- 保留任务对话区与搭配结果主舞台。
- 每个推荐商品均可进入商品详情。
- 推荐结果展示商品事实、推荐理由和 AI 建议的明确分区。
- 商品价格、可选 SKU 和库存由 Java 商品接口返回。
- 风格适配和尺码建议必须标注“AI 建议”；信息不足时显示不确定性，不输出确定结论。
- 加购前必须选择有效 SKU；已有确定 SKU 的推荐可预选，但仍允许用户修改。
- 推荐链路记录曝光、点击、加购和成交归因，不影响用户正常购买。

### 6.3 探索商品

页面采用“分类入口 + 紧凑筛选 + 单一商品网格”：

- 搜索支持商品名、分类、风格和场景关键词。
- 搜索联想分组展示关键词、分类和商品；无联想时不阻塞正常搜索。
- 筛选包含性别、品类、风格、场景、价格、尺码和库存状态。
- 排序包含综合、销量、价格和新品；选项是否出现取决于后端是否提供可信字段。
- URL 保存关键词、分类、筛选、排序和页码，支持刷新、返回和分享。
- 商品卡只显示 SPU 层摘要；颜色、尺码和库存详情进入商品详情页。
- 空结果保留当前筛选条件，并提供清除单项或全部条件的入口。

### 6.4 商品详情

桌面采用主图画廊和购买决策区双栏，手机按图片、标题价格、SKU 和购买操作顺序排列。

主要模块：

1. 面包屑和返回搜索结果。
2. 主图、缩略图、图片失败占位。
3. 商品名称、价格区间、状态和基础卖点。
4. 颜色、尺码等 SKU 选择器。
5. 选中 SKU 的真实价格、库存和可购买数量。
6. 加入购物袋、立即购买和收藏。
7. AI 建议卡：衣橱画像匹配说明、尺码建议和不确定性提示。
8. 商品详情、尺码参数、材质洗护和用户评价扩展位。
9. 相似商品和可搭配商品。

SKU 选择规则：

- 未形成完整 SKU 组合时禁用购买操作，并说明缺少的选项。
- 无库存组合不可购买，仍允许查看其他组合。
- 切换属性后重新解析 SKU，不沿用前一 SKU 的价格和库存。
- “立即购买”携带选中 SKU 和数量进入确认订单，不先写入购物袋。

### 6.5 购物袋

- 支持勾选、全选、数量修改、SKU 修改、删除和移入收藏。
- SKU 修改使用轻量选择面板，并在确认前校验价格与库存。
- 失效、下架和库存不足商品进入独立失效区，不参与合计。
- 金额区分商品小计、优惠、运费和应付金额；没有真实优惠时不显示虚假减免。
- 桌面固定右侧结算摘要；手机使用不遮挡内容的底部结算栏。
- 创建订单失败后保留用户选择和数量。

### 6.6 确认订单

- 显示本次结算商品快照，不允许在此页面任意切换到购物袋其他商品。
- 选择或新增收货地址。
- 展示配送方式、商品金额、真实优惠、运费和应付金额。
- 提交前重新校验 SKU 状态、价格和库存。
- 通过幂等键防止重复提交。
- 库存或价格变化时明确标出受影响商品，让用户确认后再提交。
- 成功后进入支付流程；失败时保留地址与结算上下文。

### 6.7 支付结果

- 只根据支付查询接口展示成功、处理中或失败。
- 成功时提供查看订单和继续购物。
- 处理中提供刷新状态，不能提前显示支付成功。
- 失败时解释可重试范围，并返回订单详情继续支付。

### 6.8 订单与履约售后

订单列表按全部、待支付、待发货、待收货、已完成和已关闭分组；仅映射后端真实状态。

订单详情包括：

- 订单状态、下一步动作和状态时间线。
- 收货地址快照、商品快照和金额明细。
- 支付信息、承运商、运单号和物流节点摘要。
- 在允许状态下显示取消、继续支付、确认收货和申请售后。
- 售后展示申请类型、原因、说明、处理状态和取消入口。

所有状态流转由 Java 服务校验，前端不通过本地改值模拟成功。

### 6.9 个人中心

个人中心首页展示账户摘要和六个子入口：

- 个人资料：头像、昵称、手机号和邮箱。
- 衣橱画像：风格、颜色、尺码、版型、场景、预算和材质偏好。
- 我的收藏：商品列表、取消收藏和进入详情。
- 收货地址：新增、编辑、删除、设为默认。
- 账户安全：修改密码、登录状态说明和退出登录。
- 我的订单：待处理订单快捷入口。

衣橱画像继续服务推荐，不与认证资料混为一张表单。手机号、邮箱和密码等敏感变更必须经过后端校验；首期不伪造验证码流程。

## 7. 管理后台设计

### 7.1 后台应用壳

- 左侧固定后台导航，顶部显示页面标题、全局搜索和管理员菜单。
- 内容区域以表格、筛选、批量操作、分区表单和少量图表为主。
- 延续水木色彩令牌，但提高信息密度，避免复用用户端杂志式商品卡作为管理组件。
- 危险操作使用二次确认，并说明影响对象和不可逆结果。
- 所有写操作展示 loading、成功或字段/全局错误。

### 7.2 数据概览

首期指标：

- 在售商品数、SKU 数和库存预警数。
- 待发货、售后处理中订单数。
- 时间范围内订单数、支付金额和成交趋势。
- 热门商品和低库存商品。

指标必须展示时间范围和口径；没有数据时显示空状态，不以随机数填充。

### 7.3 商品管理

- 按商品名、编码、分类、上下架状态和创建时间查询。
- 新增、编辑、查看、上架、下架和软删除。
- 列表展示主图、SPU 编码、分类、价格区间、SKU 数、总库存和状态。
- 商品编辑采用分步/分区表单：基础信息、图片、分类与标签、SKU 与库存、详情、上架设置。
- 离开未保存表单时提示；提交错误精确定位到对应分区。

### 7.4 分类管理

- 支持一级/二级分类、排序和启停。
- 禁止造成循环父子关系。
- 停用或删除前检查关联商品，并展示影响范围。
- 首期不实现无限层级和拖拽树编辑。

### 7.5 SKU 与库存

- SKU 由商品属性组合生成或编辑，至少支持颜色、尺码、售价、库存和状态。
- 同一商品下 SKU 编码和属性组合唯一。
- 库存调整记录调整前、调整后、原因、操作者和时间。
- 低库存阈值可按系统默认值或单 SKU 设置。
- 首期为单仓可售库存，不实现多仓调拨。

### 7.6 订单管理

- 按订单号、用户、状态、支付状态和时间范围查询。
- 查看商品、金额、地址、支付和操作记录。
- 在合法状态下执行发货、取消和售后处理。
- 发货必须填写承运商和运单号。
- 状态按钮由后端返回的可执行动作或统一状态机控制。

### 7.7 用户管理

- 按用户 ID、用户名、昵称、手机号/邮箱和状态查询。
- 查看基础资料、注册时间、订单摘要和账户状态。
- 允许启用/禁用账号，但不能直接展示密码或敏感凭据。
- 管理员无权直接修改用户衣橱画像，除非后续增加明确的客服授权流程。

### 7.8 经营分析与操作日志

经营分析首期包含订单量、成交额、转化漏斗、热门商品和分类趋势。推荐转化使用已有推荐曝光、点击、加购和成交归因数据，必须标明统计口径。

操作日志记录管理员、动作、对象类型、对象标识、结果、时间和必要的变更摘要。日志不可由普通后台写接口直接修改或删除。

## 8. 前端架构

### 8.1 目录建议

```text
frontend/src/
├── app/
│   ├── App.tsx
│   ├── router.tsx
│   ├── CustomerShell.tsx
│   └── AdminShell.tsx
├── pages/
│   ├── customer/
│   └── admin/
├── features/
│   ├── catalog/
│   ├── checkout/
│   ├── order/
│   ├── profile/
│   └── admin/
├── shared/
│   ├── api/
│   ├── repositories/
│   │   ├── contracts/
│   │   ├── http/
│   │   └── mock/
│   ├── components/
│   └── routing/
└── styles/
    ├── tokens.css
    ├── base.css
    ├── components.css
    ├── customer.css
    └── admin.css
```

页面只负责路由级布局和数据装配；业务状态和操作放入对应 feature hooks；HTTP 细节和 Mock 数据不进入页面组件。

### 8.2 路由策略

当前 `App.tsx` 使用内存 `ViewKey` 切换，无法支持商品详情深链、浏览器返回、结算恢复和独立后台路由。本轮引入成熟的 React 路由方案，建立嵌套路由、路由参数、查询条件和权限守卫。

### 8.3 状态策略

- 服务端数据由页面/feature hooks 管理，不引入全局业务状态库作为首选。
- 认证会话、购物袋数量和全局确认对话框位于应用壳上下文。
- 搜索和筛选状态进入 URL。
- 结算草稿只保存必要的 SKU、数量和地址选择；价格与库存始终重新向后端校验。
- 管理后台表格状态按页面维护，并通过 URL 保留常用查询。

### 8.4 Mock 设计

Mock 数据集至少包含：

- 有多个颜色和尺码的正常商品。
- 低库存、无库存、下架和缺图商品。
- 空购物袋、混合有效/失效购物袋。
- 待支付、待发货、待收货、完成、取消和售后订单。
- 空地址、默认地址和多地址。
- 管理端正常、空结果、字段校验失败和无权限状态。

Mock 操作在内存中可产生可见状态变化，如加购、修改 SKU、提交演示订单和后台编辑商品；刷新后是否重置必须在页面上说明。Mock 不模拟真实安全边界，也不能作为后端验收依据。

## 9. API 与后端能力规划

### 9.1 可直接复用的现有能力

- 认证：注册、登录、刷新和退出。
- 当前用户与资料：当前用户、资料、身体数据和偏好。
- 商品：列表、推荐候选和 SPU 详情查询。
- 购物袋：查询、新增、修改数量和删除。
- 订单：创建、立即购买、列表、详情和取消。
- 支付：创建、模拟支付、查询和回调。
- 售后：申请、查询和取消。
- 收藏和行为事件。

复用前仍需核对字段完整性、路径一致性和权限边界。

### 9.2 需要新增或扩展的用户端契约

## 10. 数据模型调整方向

在复用现有用户、商品、SKU、库存、订单、支付和售后表的基础上，预计增加或扩展：

- 用户收货地址及默认地址约束。
- 商品分类父子关系、排序和启停状态。
- 商品上下架、软删除和展示详情字段。
- SKU 状态、低库存阈值和属性组合唯一约束。
- 库存调整流水。
- 订单地址快照、配送信息、确认收货时间和履约节点。
- 用户账号状态与安全变更时间。
- 管理员操作日志。

数据库设计必须在后端实施前核对当前迁移和表结构，不以本文字段名直接生成迁移。金额继续使用精确十进制；库存更新使用现有并发控制和幂等规则。

## 11. 权限与安全

- `/app/*` 需要登录，公开商品浏览是否开放由后续产品决策决定；首期沿用当前登录后访问。
- `/admin/*` 需要管理员角色，Java 后端对每个管理接口执行授权。
- 管理员禁用用户、调整库存、上下架、发货和售后处理均写审计日志。
- 密码永不返回前端；修改密码需要校验旧密码或后续接入可信验证流程。
- 图片地址、富文本详情和搜索条件需要服务端校验与输出安全处理。
- 前端不在日志、URL 和持久化状态中保存访问令牌以外的敏感信息；继续沿用并评估现有令牌存储策略。

## 12. 加载、错误和一致性

所有新增页面至少具备加载、空状态、错误和重试四类状态。

- 商品详情获取失败时不显示可购买按钮。
- SKU/库存校验失败时保留选择，并指出失效组合。
- 结算失败时保留地址和结算上下文。
- 管理表单字段错误就地显示，全局错误保留所有已输入内容。
- 删除、下架、发货、取消和库存调整等写操作防止重复提交。
- 前端乐观更新仅用于可安全回滚的弱一致操作；订单、支付、库存和后台状态流转以服务端响应为准。

## 13. 响应式与可访问性

- 验收宽度继续使用 1440px、900px 和 390px。
- 商品详情在窄屏变为单列，并提供不遮挡内容的购买操作区。
- 筛选在手机端进入抽屉，已选条件在结果区可见且可逐项移除。
- 确认订单在手机端将金额摘要置于提交按钮附近。
- 管理后台在 900px 保持可操作；390px 允许查看关键数据和处理简单动作，但复杂商品编辑优先提示使用宽屏。
- 保证键盘导航、可见焦点、语义标题、表格标题、表单标签和弹窗焦点管理。
- 颜色不是状态的唯一表达，动效遵守 `prefers-reduced-motion`。

## 14. 测试策略

### 14.1 前端阶段

- 类型和仓储契约测试：Mock 与 HTTP 映射遵循同一领域类型。
- 组件测试：SKU 选择、金额展示、地址表单、状态动作和管理表单校验。
- 页面测试：加载、空、错、无权限和成功状态。
- E2E 用户链路：搜索/AI → 商品详情 → 选择 SKU → 购物袋/立即购买 → 确认订单 → 支付结果 → 订单详情。
- E2E 用户资产：资料、衣橱画像、收藏和地址。
- E2E 管理链路：新增商品 → 配置 SKU/库存 → 上架 → 查询订单 → 发货。
- 视觉验收：1440、900、390 三档截图。
- 构建验证：`npm test -- --run`、`npm run test:e2e`、`npm run build`。

### 14.2 后端与联调阶段

- Controller、Service、Mapper 和数据库迁移测试。
- 管理权限、状态机、库存并发、订单幂等和审计测试。
- 前后端契约测试，覆盖字段缺失、错误码和分页。
- 真实 API 模式重跑用户端和管理端 E2E。
- Java 完整验证：`backend\mvnw.cmd verify`。

## 16. 前端效果验收标准

进入后端开发前，前端必须满足：

1. 用户可以在 Mock 模式完成从首页或 AI 推荐到支付结果、订单详情和售后的完整演示。
2. 每个商品必须经过详情页或明确的 SKU 决策步骤后购买。
3. 个人中心完整展示资料、衣橱画像、收藏、地址和安全子页面。
4. 管理员可以演示新增商品、编辑 SKU/库存、上下架、订单发货和用户查询。
5. 用户端与管理端路由和应用壳明确分离。
6. 1440px、900px 和 390px 下关键路径无横向溢出或操作遮挡。
7. 加载、空、错、无权限和危险操作确认状态完整。
8. Mock 模式始终可辨认，API 模式不回退到演示事实。
9. 现有 AI 聊天、推荐、购物袋、订单和偏好测试不因改版失效。
10. 前端单元测试、E2E 和生产构建全部通过。

## 17. 风险与控制

| 风险 | 控制措施 |
| --- | --- |
| 前端先行导致契约返工 | 先定义 repository 和 TypeScript DTO，Mock 与 HTTP 共用契约测试。 |
| Mock 被误认为真实数据 | 显式环境开关、全局演示标记、正式构建默认 API 模式。 |
| 页面一次性过多 | F0–F3 分阶段，每阶段保持可运行、可测试、可截图验收。 |
| 当前 `App.tsx` 继续膨胀 | 引入路由和双壳层，将业务状态下沉到 feature hooks。 |
| 后台权限只做前端隐藏 | 所有管理 API 在 Java 层校验管理员角色并审计。 |
| 状态命名与现有后端不一致 | UI 文案映射与原始状态分离，逻辑始终使用后端枚举。 |
| 库存和金额被前端篡改 | 结算、下单和库存操作只采信 Java 重新计算的结果。 |
| 改版破坏现有推荐能力 | 保留 Java/Python 边界和行为埋点，按阶段运行现有测试。 |
