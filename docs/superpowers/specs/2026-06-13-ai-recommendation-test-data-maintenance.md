# AI 推荐测试数据维护方案

日期：2026-06-13

## 1. 文档目的

本文档只讨论 AI 推荐页当前阶段需要维护和补充的数据库测试数据，不涉及代码修改。

当前 AI 推荐链路已经具备以下能力：

- Java 从商品库加载候选商品并传给 Python。
- Python 根据用户意图、尺码工具、风格词、价格和库存生成 `product_refs`。
- Java 只接受来自当前候选池的 `spuId + skuId`，防止 Python 编造商品。
- 前端根据 `recommendedSpuIds` 对推荐卡片排序。

因此，数据库数据质量会直接决定 AI 推荐验证是否可信。当前种子数据只有 3 个 SPU、11 个 SKU，能够验证链路是否跑通，但不足以验证推荐是否准确、稳定、可解释。

## 2. 当前数据不足带来的问题

### 2.1 尺码覆盖不完整

当前 `size_option` 只有 `S/M/L/XL`，种子商品虽然有部分 XL SKU，但关键 XL SKU 库存可能为 0。

这会导致：

- 用户问 `177 130 怎么穿？`，尺码工具正确推荐 XL。
- 但 Java 候选池没有可售 XL 代表商品。
- Python 不能返回有效 `product_refs`。
- 前端看起来像是“AI 没推荐商品”。

这个现象不是单纯的模型问题，而是商品库无法承接尺码结果。

### 2.2 SPU 数量太少，无法验证排序

当前只有：

- T恤：基础款纯棉T恤
- 外套：通勤轻薄外套
- 长裤：直筒休闲长裤

这只能验证“是否能展示商品”，不能验证：

- 面试 vs 日常 vs 运动场景是否会改变排序。
- 显瘦 vs 宽松 vs 修身偏好是否会改变排序。
- 夏季 vs 冬季是否会改变候选。
- 预算 100、300、500、800 是否会改变结果。
- 库存不足或下架商品是否会被避开。

### 2.3 缺少负样本和边界样本

当前数据基本都是可售正样本，缺少：

- 无库存 SKU。
- 下架 SKU。
- 价格超预算商品。
- 风格不匹配商品。
- 尺码不匹配商品。
- 同 SPU 多 SKU 中部分有库存、部分无库存的情况。

没有这些数据，就无法验证过滤逻辑是否真的在工作。

### 2.4 当前候选 SQL 的代表 SKU 风险

`ProductMapper.findRecommendationCandidates` 当前按 SPU 聚合，并使用：

- `MIN(sku.id) AS sku_id`
- `MIN(co.name) AS color`
- `MIN(so.code) AS size`
- `MIN(sku.sale_price) AS sale_price`

这意味着候选列表里每个 SPU 只暴露一个代表 SKU。补数据时需要特别注意：

- 如果一个 SPU 同时有 M/L/XL，但 `MIN(sku.id)` 对应的是 M，那么 Python 看到的候选尺码可能就是 M。
- 用户问 XL 时，即使这个 SPU 真实存在 XL SKU，Python 也可能因为候选代表 SKU 是 M 而不推荐。
- 这会让“商品真实有货”与“AI 候选可见信息”不一致。

现阶段补测试数据时，建议使用两种策略：

- 短期：为关键测试尺码准备独立 SPU，确保代表 SKU 的尺码就是要验证的尺码。
- 长期：后续再讨论候选 SQL 是否应该按 SKU 维度返回，而不是按 SPU 聚合。

## 3. 数据维护目标

当前阶段数据目标不是追求真实电商规模，而是构造一套能稳定验证 AI 推荐能力的最小充分数据集。

建议目标：

- SPU：至少 24 个，推荐 36 个。
- SKU：至少 80 个，推荐 120 个。
- 每个核心品类至少覆盖 `S/M/L/XL/2XL`。
- 每个关键场景至少有 2 个可推荐商品、1 个干扰商品、1 个负样本。
- 每个价格段至少有 2 个商品。
- 每个库存状态至少有明确样本。

## 4. 必须补充的基础字典

### 4.1 尺码

当前已有：

- S
- M
- L
- XL

建议补充：

- 2XL
- 3XL

原因：

- `177 130` 只能覆盖 XL。
- 还需要测试更大体型，如 `185 170`、`190 190`。
- 没有 2XL/3XL，AI 只能回答尺码，但无法推荐对应商品。

### 4.2 风格标签

当前已有：

- commute：通勤
- casual：休闲
- minimal：极简
- sport：运动

建议补充：

- interview：面试
- slimming：显瘦
- date：约会
- outdoor：户外
- warm：保暖
- breathable：透气
- formal：正式
- vacation：度假

原因：

- 当前用户常问“面试”“显瘦”“通勤”“预算”。
- 没有这些标签，AI 的风格排序只能靠商品名或描述猜。

### 4.3 材质

当前已有：

- 纯棉
- 聚酯纤维混纺
- 棉涤混纺

建议补充：

- 羊毛混纺
- 牛仔
- 亚麻
- 莫代尔
- 锦纶
- 防晒面料
- 羽绒/仿羽绒

原因：

- 用户会问“夏天透气”“冬天保暖”“面试有质感”“不要皱”。
- 材质是解释推荐理由的重要事实字段，不能让 AI 编。

### 4.4 品类

当前已有：

- T恤
- 外套
- 长裤

建议补充：

- 衬衫
- 西装/西装外套
- 卫衣
- 牛仔裤
- 半身裙或连衣裙
- 短裤
- 针织衫

当前阶段最少应补：

- 衬衫
- 西装外套
- 牛仔裤
- 卫衣

原因：

- 面试和通勤场景只靠 T恤、外套、长裤不够。
- 用户问“怎么穿”时，AI 应该能推荐上衣、外套、裤装的组合候选。

| 品类 | 最少 SPU | 推荐 SPU | 主要验证内容 |
| --- | ---: | ---: | --- |

### 5.2 每个关键 SPU 的 SKU 结构

每个核心 SPU 建议至少有：

- 3 个尺码：M/L/XL
- 2 个颜色：黑色/白色或藏青色/灰色
- 每个 SKU 独立库存

关键验证 SPU 建议覆盖：

- S/M/L/XL/2XL 全尺码。
- 至少一个颜色有完整尺码。
- 至少一个颜色缺货，用于验证库存。

| 宽松休闲卫衣 | 灰色 | M/L/XL/2XL | L/XL/2XL 有货 |

## 6. 必须覆盖的测试场景数据

### 6.1 尺码推荐场景

| 用户输入 | 期望尺码 | 数据要求 | 验证点 |
| --- | --- | --- | --- |
| `160 60kg呢？` | M，备选 L | M 有商品，L 也有商品 | 当前轮覆盖历史，主尺码优先，备选尺码可兜底 |
| `177 130 怎么穿？` | XL | 至少 3 个可售 XL 商品 | 不再出现尺码正确但无商品可推 |
| `185 170 怎么选？` | 2XL 或 3XL | 2XL/3XL 商品必须存在 | 大体型推荐不空 |
| `155 45kg 穿什么？` | S | S 商品必须存在 | 小尺码推荐不空 |
| `我想宽松一点` | 当前尺码 +1 | 同 SPU 或同类商品有大一码 | 偏好影响推荐 |

### 6.2 多轮上下文场景

| 对话 | 数据要求 | 验证点 |
| --- | --- | --- |
| 第一轮 `177 130 怎么穿？`，第二轮 `160 60kg呢？` | XL、M/L 都有商品 | 第二轮不能复用第一轮尺码 |
| 第一轮 `明天面试`，第二轮 `预算 300 以内` | 面试商品中有 300 内和 300 外 | 预算收窄后重排 |
| 第一轮 `想显瘦`，第二轮 `不要太正式` | 显瘦正式、显瘦休闲都有 | 风格偏好变更后重排 |
| 第一轮 `夏天通勤`，第二轮 `换成冬天` | 夏季和冬季通勤商品都有 | 季节条件覆盖 |

### 6.3 风格排序场景

| 用户输入 | 应命中标签 | 数据要求 |
| --- | --- | --- |
| `明天面试，想显瘦，预算 500 内` | interview + slimming + formal/commute | 至少 2 个强匹配，1 个弱匹配，1 个超预算 |
| `日常穿，舒服一点，200 以内` | casual + breathable/regular | 至少 3 个 200 内商品 |
| `夏天通勤，不要太热` | commute + summer + breathable | 夏季通勤商品必须存在 |
| `冬天上班，保暖一点` | commute + winter + warm | 冬季保暖商品必须存在 |
| `运动休闲，预算 300` | sport + casual | 运动休闲商品必须存在 |

### 6.4 预算场景

至少准备以下价格段：

| 价格段 | 商品数量 | 用途 |
| --- | ---: | --- |
| 99-149 | 4 | 低预算、基础款 |
| 150-299 | 8 | 日常、通勤基础 |
| 300-499 | 8 | 面试、质感、外套 |
| 500-799 | 4 | 高质感、冬季、正式 |
| 800+ | 2 | 超预算负样本 |

验证问题：

- `预算 200 以内`
- `预算 500 内`
- `贵一点有质感`
- `便宜点`

### 6.5 库存和上下架场景

必须有：

- 可售且库存充足：`available_stock >= 8`
- 低库存：`available_stock = 1 或 2`
- 无库存：`available_stock = 0`
- SKU 下架：`product_sku.status != 'on_sale'`
- SPU 下架：`product_spu.status != 'on_sale'`

验证目标：

- AI 不应该推荐无库存或下架商品。
- Java 候选池不应该把无库存商品传给 Python。
- 低库存可以展示，但排序应低于库存充足商品。
- 加购/购买最终仍以 Java 库存校验为准。

### 8.1 单轮验证

| 问题 | 期望 |
| --- | --- |

| `160 60kg呢？` | 回答 M，必要时推荐 L 备选商品 |

| `日常穿，舒服点，200 以内` | 低价基础款、休闲款靠前 |
| `运动休闲，预算 300` | 运动 T 恤、运动裤、防晒外套靠前 |

| 对话 | 期望 |
| --- | --- |

| 问题 | 期望 |
| --- | --- |
| `预算 100 以内买西装外套` | 如果无匹配，应明确说明没有合适候选，不编商品 |

## 9. 数据维护规则

### 9.1 ID 和命名

建议为补充测试数据单独预留 ID 段：

- SPU：`3001-3999`
- SKU：`300001-399999`
- 图片：继续使用 `/images/products/...`

命名建议：

- `TSHIRT_SUMMER_BREATHABLE_001`
- `SHIRT_INTERVIEW_ANTIWINKLE_001`
- `BLAZER_SLIMMING_COMMUTE_001`
- `PANTS_FORMAL_STRAIGHT_001`

这样日志里看到 `spu_code` 和 `sku_code` 时能直接判断用途。

### 9.2 标签必须真实反映商品

不要为了让测试通过乱贴标签。

例如：

- 面试衬衫可以贴 `interview, commute, formal, minimal`。
- 运动 T 恤不应该贴 `formal`。
- 羊毛大衣可以贴 `warm, winter, commute`。
- 夏季速干 T 恤不应该贴 `winter`。

AI 推荐的可解释性依赖这些事实字段。

### 9.3 每个测试场景都要有对照组

每个场景至少包含：

- 强匹配商品：应该排前面。
- 弱匹配商品：可以出现但不应排第一。
- 干扰商品：品类或风格相近，但条件不完全匹配。
- 负样本：预算、库存、尺码或状态不满足，不应出现。

没有对照组，就无法判断排序是否真的有效。

### 9.4 库存不要全部设置成满库存

建议库存分布：

- 充足：`available_stock = 10-30`
- 低库存：`available_stock = 1-2`
- 无库存：`available_stock = 0`
- 已锁定库存：少量 SKU 设置 `locked_stock > 0`，用于后续订单链路测试

### 9.5 价格不要过于集中

同一场景内需要有价格梯度。

例如面试场景：

- 199：抗皱衬衫
- 299：西装裤
- 499：西装外套
- 799：羊毛混纺大衣
- 1299：超预算负样本

这样才能测试“预算 300”“预算 500”“贵一点有质感”。

## 10. 验收标准

补数完成后，至少满足以下标准：

- `177 130 怎么穿？` 能得到非空推荐商品。
- `160 60kg呢？` 能按 M 主尺码或 L 备选尺码返回商品，且不会复用上一轮 177/130。
- `明天面试，想显瘦，预算 500 内` 能返回至少 2 个强匹配商品。
- `夏天通勤，不要太热` 能返回夏季、透气、通勤相关商品。
- `冬天上班，保暖一点` 能返回冬季、保暖相关商品。
- 预算明显不满足时，不返回超预算商品。
- 无库存或下架商品不进入 AI 推荐候选。
- 前端推荐卡片排序能随不同问题发生变化。

## 11. 当前优先级

建议按下面顺序补：

1. 补 `2XL/3XL` 尺码字典。
2. 补 `interview/slimming/warm/breathable/formal` 等核心风格标签。
3. 补面试/通勤组商品，优先解决 `177 130 -> XL 有商品可推`。
4. 补日常/夏季组商品，覆盖 `160 60kg` 和低预算问题。
5. 补库存负样本和超预算负样本。
6. 补秋冬/运动/户外组，用于扩大推荐覆盖面。

现阶段最重要的是先让 AI 推荐有足够候选可选。否则继续调模型或排序规则，很容易把“数据不够”误判成“算法不准”。
