# 水木 AI 全站前端结构级改版设计

## 1. 背景与目标

现有 React + TypeScript + Vite 前端已经完成一轮 NOIR.AI 深色未来风格改版，包括顶部导航、AI 左侧对话与右侧推荐、深色商品卡和全站主题统一。下一轮改版不能只将深色替换为绿色或木色，否则页面结构、使用路径与产品体验不会产生足够价值。

本轮采用“浅橡木自然”视觉方向和“全站结构级改版”路线：保留现有后端接口与业务流程，重做应用壳、登录后落地页、信息层级、页面布局、组件外观和响应式阅读顺序，让长期使用的真实用户获得更清晰、自然且具有识别度的穿搭购物体验。

### 目标

- 以鼠尾草绿、亚麻白和浅橡木建立自然、克制、耐看的全站视觉系统。
- 将顶部导航改为稳定的侧边应用导航；窄屏降级为底部导航。
- 新增“我的穿搭空间”首页，承接已有用户状态、偏好、最近任务和主要入口。
- 将 AI 导购升级为连续的穿搭任务工作区，使搭配结果成为主舞台。
- 按探索、决策、跟踪和画像维护四类任务重做商品、购物袋、订单和偏好页面。
- 保留现有认证、聊天、推荐、搜索、加购、下单、支付和偏好保存行为。
- 建立桌面、平板、手机三档可验证的质量基线。

### 不在范围内

- 不修改 Java 后端、Python AI 服务、共享契约或 API 字段。
- 不新增试穿、收藏、天气服务、搭配保存、物流或真实支付能力。
- 不伪造天气、价格、库存、推荐理由、匹配分、订单状态或用户画像。
- 不引入新的 UI 组件库、3D 引擎、Canvas 粒子系统或重型动效依赖。
- 不将本轮扩展为产品流程重构；聊天到推荐、商品到订单的业务状态机保持不变。

## 2. 已确认决策

| 决策 | 结论 |
| --- | --- |
| 使用场景 | 面向真实用户长期使用，不以答辩演示冲击力为唯一目标。 |
| 视觉方向 | 浅橡木自然：鼠尾草绿、亚麻白、浅木色，科技感隐藏在 AI 反馈与信息组织中。 |
| 改版范围 | 登录、首页、AI 造型师、探索商品、购物袋、订单、衣橱画像全站统一升级。 |
| 改版深度 | 结构级改版，不接受仅替换颜色、阴影和圆角的主题换肤。 |
| 导航 | 桌面使用左侧应用导航，窄屏使用底部导航。 |
| 登录后入口 | 新增“我的穿搭空间”首页，不再直接进入现有 AI 双栏页面。 |
| AI 页面 | 对话成为任务区域，搭配结果成为主舞台；状态仅由现有数据推导。 |
| 业务边界 | 保留现有接口、请求时机和行为语义，不绕过 Java 后端数据边界。 |

## 3. 全站信息架构

登录后的应用由稳定壳层和六个主视图组成：

```text
水木 AI 应用壳
├── 今日：首页 / 我的穿搭空间
├── AI 造型师：聊天、需求线索和推荐搭配
├── 探索商品：搜索、筛选和商品结果
├── 购物袋：商品管理和创建订单
├── 订单：订单状态和下一步动作
└── 衣橱画像：用户偏好维护
```

桌面侧边栏固定显示品牌、主导航、用户入口和会话状态。平板可收窄为图标栏；手机改为底部主导航，并将低频用户设置放入顶部个人入口。导航变化不改变现有前端视图状态和 API 调用边界。

### 我的穿搭空间

首页负责承接登录后的状态，而不是复制商品中心或 AI 页面：

- 顶部问候和主要 AI 入口。
- 使用已有偏好和最近前端状态形成轻量“衣橱信号”。
- 使用已有推荐结果展示今日灵感；没有结果时展示进入 AI 造型师的引导。
- 展示购物袋数量和待处理订单入口等已有数据。
- 天气没有现成可信接口，本轮不展示真实天气；视觉草图中的天气仅代表未来可选扩展，不进入实现验收。

## 4. 页面设计

### 4.1 登录页

- 从深色玻璃登录框改为亚麻白背景、浅木质分区和绿色主操作。
- 宽屏采用品牌叙事区与登录表单双区；窄屏合并为单列。
- 保留登录、注册、校验、错误信息和 loading 行为。
- 登录成功后直接进入“我的穿搭空间”，不播放独立启动页。

### 4.2 AI 造型师

页面采用任务区域与搭配画布组成的工作区：

- 任务区域承载现有聊天消息、流式回复、输入框和需求线索。
- 需求线索只从用户消息、现有筛选条件和推荐元数据中提取；无法确认的值不显示。
- “描述需求 → AI 理解 → 查看搭配”仅是展示阶段，由当前聊天和推荐状态推导，不新增后端状态机。
- 搭配画布将第一项推荐作为主单品，第二和第三项作为配套单品，其余候选继续在后续结果区展示。
- 推荐理由、总价、库存和操作集中在画布底部；总价只对拥有真实价格的展示项求和。
- “加入购物袋”仍逐项调用现有加购行为，不引入后端不支持的整套原子加购语义。
- 没有推荐时展示需求示例和空画布；加载时显示稳定骨架；失败时保留错误和重试入口。

### 4.3 探索商品

- 移除当前 SPU 摘要列表与 SKU 候选卡片的重复视觉层级。
- 页面以搜索、紧凑筛选和单一商品策展网格组成。
- 移除重复出现的聊天侧栏；需要 AI 帮助时通过明确入口跳转到 AI 造型师。
- 继续使用现有商品搜索和推荐候选接口；若业务必须保留 SPU 与 SKU 两类结果，则通过单卡详情层级呈现，不在同一首屏重复展示。
- 桌面使用不等宽策展网格，手机使用单列卡片并保持价格、库存和操作可见。

### 4.4 购物袋

- 桌面使用“商品管理 + 固定订单摘要”双区结构。
- 左侧保留选择、数量调整、删除、图片、规格和单价。
- 右侧展示已选数量、真实合计金额和创建订单操作。
- 手机端订单摘要收敛为底部固定结算栏，并为安全区域留出空间。
- 创建订单、库存失败和网络失败沿用现有接口结果，失败后不清空用户选择。

### 4.5 订单

- 将状态、时间和下一步操作提升为主要信息，弱化纯卡片堆叠。
- 可按现有状态在前端分组显示全部、待支付、进行中和已完成；不为不存在的状态制造入口。
- 支付按钮只在后端返回可支付状态时出现。
- 刷新能力保留，但作为次级操作。
- 订单状态文案通过统一映射呈现，原始状态值仍用于逻辑判断和测试。

### 4.6 衣橱画像

- 保留现有偏好字段和保存接口，按“风格与颜色、尺码与版型、日常场景、预算与材质”重组。
- 每组解释这些信息如何帮助推荐，避免后台管理表单观感。
- 完成度仅根据现有字段的已填写情况计算；不新增隐私字段或推断用户敏感属性。
- 保存状态、字段错误和全局错误分别显示，避免用户无法判断失败位置。

## 5. 设计系统

### 5.1 语义令牌

所有页面使用 CSS 自定义属性，不在组件中散落具体色值。

| 用途 | 建议令牌 | 基准色 |
| --- | --- | --- |
| 深色导航与强对比 | `--color-forest-900` | `#263A2D` |
| 品牌主色与主要状态 | `--color-sage-600` | `#78936D` |
| 页面基础背景 | `--color-linen-50` | `#F6F1E8` |
| 木质强调与图片衬底 | `--color-oak-400` | `#BDA98B` |
| 轻状态与标签背景 | `--color-moss-100` | `#DCE8D6` |
| 主文字 | `--text-primary` | 深森林灰绿 |
| 次级文字 | `--text-secondary` | 中性暖灰绿 |
| 错误 | `--status-danger` | 可读的低饱和砖红 |
| 警告 | `--status-warning` | 可读的暖赭色 |

木纹仅可通过低对比 CSS 渐变或小型本地纹理表达，不能覆盖正文区域或引入影响首屏性能的大图。

### 5.2 字体与图标

- 中文正文继续使用可靠的系统无衬线字体栈。
- 重要页面标题可少量使用系统宋体/衬线字体栈，形成自然编辑气质；表单、数字和操作保持无衬线。
- 英文小标签使用适度字距，不以全大写堆叠制造噪声。
- 继续复用 Lucide 图标，建立 16、18、20px 三档尺寸。
- 关键操作不能只靠图标表达，必须保留文本或可访问名称。

### 5.3 组件策略

- 建立共享 `AppShell`、`SideNavigation`、`MobileNavigation`、`PageHeader`、`StatusBanner` 和基础表面组件。
- 业务组件继续位于既有特性目录；共享视觉组件不能承接业务状态。
- `ProductCard` 保留统一数据和操作接口，通过展示变体支持策展主卡、配套卡和标准商品卡。
- 表单控件、按钮、标签、骨架、空状态和错误状态使用共享语义 class 或轻量组件。
- 不引入全局状态库；现有 `App` 与特性 hooks 继续拥有业务状态。

## 6. 动效与反馈

- 按钮、筛选和卡片反馈使用 160–220ms，最多产生 2px 位移。
- 页面进入使用 360–520ms，按标题、主区域和辅助卡轻量错峰显现。
- AI 流式回复与推荐加载使用柔和扩散、光标或叶脉式进度线，不使用大面积荧光扫描和粒子。
- 导航切换不重复播放整站启动动画。
- 所有操作必须即时呈现 loading、成功或失败反馈，不能等待动画结束才执行。
- `prefers-reduced-motion: reduce` 下关闭位移、错峰和循环动画。

## 7. 数据流与业务边界

```text
认证成功
  -> App 保留用户、视图、购物袋、AI 与确认动作状态
  -> AppShell 导航到“我的穿搭空间”
  -> 首页消费已有偏好、推荐、购物袋和订单摘要
  -> AI 造型师复用 ChatPanel 流式请求与推荐回调
  -> 推荐数组按后端顺序映射为主单品、配套单品和后续结果
  -> ProductCard / 购物袋 / 订单继续调用现有 commerce API
  -> 衣橱画像继续读取与保存现有偏好字段
```

Java 后端仍是用户、商品、SKU、价格、库存、订单和支付状态的唯一可信来源。Python AI 服务仍通过既有 Java 边界提供推荐相关能力。前端不得自行生成业务事实，也不得绕过 Java 服务直接拼装可信上下文。

首页是新的展示组合层，不拥有新的业务数据。若需要的数据在现有接口中不存在，首期实现必须隐藏对应模块或使用明确的空状态，而不是补造示例值。

## 8. 加载、空状态与错误处理

每个主页面必须具备四类明确状态：

1. 加载：使用与最终布局近似的稳定骨架，避免跳动。
2. 空状态：解释为什么为空，并提供一个与当前任务相关的下一步入口。
3. 错误：显示可理解的错误信息，保留现有用户输入和选择。
4. 重试：仅重试失败请求，不重置无关页面状态。

图片加载失败时使用统一的水木品牌占位，不能通过隐藏 `<img>` 留下空白区域。推荐元数据缺失时显示“AI 精选”而非伪造匹配百分比；推荐理由缺失时只展示真实商品属性。

## 9. 响应式与可访问性

- 基准验收宽度为桌面 1440px、平板 900px、手机 390px。
- 桌面侧边栏在平板收窄为图标栏，手机改为底部导航。
- AI 页面在窄屏按“任务与对话 → 搭配结果”顺序排列，输入区和主要操作保持可达。
- 购物袋在窄屏使用底部结算栏，并避免遮挡最后一项内容。
- 保证键盘导航、可见焦点、语义标题、表单标签、按钮名称和弹窗焦点管理。
- 正文、次要文字、边框和状态色满足可读对比，颜色不是唯一的状态表达。

## 10. 实现边界

建议以现有特性目录为基础演进：

| 区域 | 责任 |
| --- | --- |
| `src/app/App.tsx` | 保留顶层业务状态；接入新首页与壳层，不继续堆积页面视觉结构。 |
| `src/app/AppShell.tsx` | 新应用壳、桌面侧栏、移动导航和页面内容容器。 |
| `src/pages/HomePage.tsx` | 聚合已有数据形成登录后首页，不拥有业务事实。 |
| `src/pages/AiShoppingPage.tsx` | 重组为任务区域和搭配画布。 |
| `src/pages/ProductBrowsePage.tsx` | 单一探索画布、搜索和筛选结果。 |
| `src/pages/CartPage.tsx` | 商品管理和固定结算摘要。 |
| `src/pages/OrdersPage.tsx` | 状态映射、分组和下一步动作。 |
| `src/pages/ProfilePreferencesPage.tsx` | 按画像语义重组现有字段。 |
| `src/features/catalog/ProductCard.tsx` | 共享商品业务行为与三类展示变体。 |
| `src/styles.css` | 逐步拆分为令牌、基础、壳层和页面样式；不引入 CSS-in-JS。 |

为控制风险，样式可按 `styles/tokens.css`、`styles/base.css`、`styles/components.css` 和页面局部 CSS 文件拆分；具体拆分由实施计划根据当前构建配置确认。

## 11. 测试与验收

### 自动验证

- 运行 `npm run build`，验证 TypeScript 和 Vite 构建。
- 运行 `npm test`，保留现有聊天状态、商品操作、API 和偏好测试。
- 运行 `npm run test:e2e`，覆盖登录、导航、AI 对话、推荐、加购、创建订单、支付和偏好保存主流程。
- 补充新首页、桌面/移动导航、页面空状态、错误状态和 390px 关键路径测试。
- 对 `prefers-reduced-motion` 进行至少一条自动或人工验证。

### 人工验收

- 1440px 下侧边栏、首页、AI 工作区和购物袋双区结构完整，无横向溢出。
- 900px 下导航收窄，主要内容不被压缩到不可读。
- 390px 下底部导航、聊天输入和购物袋结算栏互不遮挡。
- 全站不再保留 NOIR 深色主题残留，也不只是替换背景颜色。
- 商品图、价格、库存、推荐理由和订单状态均来自真实数据。
- 加载、空、错、重试状态视觉统一且不丢失用户操作上下文。
- 键盘可完成导航、聊天、筛选、加购、结算和偏好保存。

## 12. 开发前准备

1. 锁定页面清单：登录、首页、AI 造型师、探索商品、购物袋、订单、衣橱画像。
2. 盘点现有接口字段，建立首页和衣橱画像展示项到真实数据源的映射；没有来源的项从首期移除。
3. 检查真实商品图片的比例、清晰度和缺图率，确定统一图片容器和本地占位资产。
4. 建立 1440px、900px、390px 三档基准截图目录和人工验收表。
5. 标记现有端到端测试依赖的 `data-testid`，改结构时保持或有计划地迁移。
6. 在不覆盖当前未提交后端改动的前提下，从新的前端功能分支开始实施。

## 13. 实施顺序建议

1. 设计令牌、基础组件和响应式壳层。
2. 登录页与新的“我的穿搭空间”首页。
3. AI 造型师任务区与搭配画布。
4. 探索商品和共享商品卡变体。
5. 购物袋、订单和衣橱画像。
6. 全站空状态、错误、动效、无障碍和三档视觉验收。

每个阶段都应保持应用可构建、现有业务测试可运行，避免先完成全站静态样式后再集中恢复业务行为。
